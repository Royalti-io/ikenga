The full queue, end to end.

Screen 04 dialed in the approval row. This screen is about everything around it — the four channels (Email / Newsletter / Sequences / Social) that share one Outbox sidebar, the three views each channel exposes (Queue · Schedule · Sent), the action ladder a draft walks (edit → save → schedule → approve → send → log), and the Claude session hand-off: send the draft into chat for a quick rewrite, continue an existing session, or spawn a fresh one with the draft as seed prompt.

AChannels in the sidebar · views in the content
/outbox owns four channels — Email, Newsletter, Sequences, Social — surfaced as first-class entries in the Outbox sidebar's Channels group. There is no outer channel tab strip; the sidebar already tells you which channel is active, and a duplicate row of tabs would just compete for attention. Inside each channel, three views are pinned to the top of the content body: Queue (drafts awaiting), Schedule (calendar of approved + cooling), Sent (history + per-channel charts). There is no outer "Sent" tab either — sending happens by channel, so reading sent does too.

/outbox/<channel>/* · view tab strip (in-content)
three views · same grammar in every channel
Approval queue · master/detail of drafts in draft/pending_review. Each channel renders its channel-specific detail pane (Newsletter: cooling banner + scorecard; Email: subject/body editor; Social: per-platform char counters; Sequences: step preview).
Schedule · calendar of approved + cooling sends. Newsletter uses a 3-week Lagos-TZ strip; Email/Social use the same grid; Sequences swaps the calendar for an "Active runs" timeline (cohorts × steps), since sequences fire on a per-recipient cadence rather than a single timestamp.
Sent · history table by default; ?view=charts toggles to per-channel charts (engagement / deliverability / drilldown). The smart-landing redirect on /outbox/<channel> sends you to the most useful view: ready drafts → Approval queue, only cooling → Approval queue?focus=cooling, nothing pending → Schedule, on a charts day → Sent?view=charts.
BThe draft's life · 6 steps from agent to log
Every draft (email, newsletter section, sequence step, social post) walks the same ladder. The Claude session hand-off lives between edit and approve — it's the escape hatch when in-place editing isn't enough. After send, the row moves to /outbox/<channel>/sent and the Supabase row gets a sent timestamp + provider message-id.
Draft lifecycle · status transitions
drafted → edited → scheduled → approved → sending → sent
1 · Drafted

Agent writes into email_drafts with status='draft'. Lands in /outbox/email.

2 · Edit in place

Subject input + body textarea autosave. ⌘S saves; status stays draft but row gets edited tick.

3 · Hand-off (optional)

Stuck on tone? ⌘K sends draft + context to dock chat. Or "Continue session" opens the Claude session that wrote it.

4 · Schedule

Pick a send time (default: agent-suggested). Status → scheduled. Quiet-hours rules in /settings auto-shift weekend sends.

5 · Approve & send

⌘↵ flips to approved. The actions sidecar's poller picks it up and dispatches via Resend / Listmonk / SMTP / Buffer.

6 · Sent + logged

Provider message-id + timestamp persisted. Row moves to /outbox/<ch>/sent. Webhooks (Resend) update bounce/open later.

Newsletter fork · the newsletter pipeline inserts a cooling step between drafted and edit. The Approve button is locked for 2h after queue-insert (reviewable_after = now + 2h) so you actually read the draft. See screen 11 §C for the live countdown UI.

HClaude session hand-off · three destinations from one draft
In-place editing covers ~80% of approvals. The other 20% — tone rewrites, fresh-context regeneration, "this isn't what I meant" — needs a real conversation. Three escape hatches, all reachable from any draft's footer.

KRound-trip · pulling rewrites back into the draft
Hand-off is one direction; the return trip matters more. Inside any chat thread (dock or /sessions/<id>), Claude messages get an "Apply to draft" affordance when the originating context was a draft. Click it → the draft's body is replaced (with old version stashed in prior_versions[] for undo) and the user is bounced back to /outbox/email/<id> with the new body shown + a one-click revert.
